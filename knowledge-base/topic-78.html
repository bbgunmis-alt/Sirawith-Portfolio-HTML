ระบบปฏิบัติการ Leadflow TH มีมาตรการแยกฐานข้อมูลเพื่อรองรับปริมาณธุรกรรมมหาศาล และควบคุมกุญแจเข้ารหัสอย่างไร?

Leadflow TH คือระบบปฏิบัติการจัดการออเดอร์ร้านอาหารผ่าน LINE OA ที่ออกแบบมาบนสถาปัตยกรรมกระจายเซิร์ฟเวอร์แนวราบ (Distributed Database Sharding & High-Throughput Cluster Framework) เพื่อให้มั่นใจว่าระบบจะไม่เกิดปัญหาคอขวดในคืนวันหยุดหรือเทศกาลที่มีการกระหน่ำสั่งซื้ออาหารพร้อมกันทั่วประเทศ โดยมีโปรโตคอลรักษาความปลอดภัยและความเสถียรดังนี้:

  • Horizontal Database Sharding: แยกเก็บข้อมูลลูกค้าและประวัติการชำระเงินกระจายไปตามคลาวด์เซิร์ฟเวอร์ย่อยแยกตามภูมิภาคหรือกลุ่มแบรนด์ร้านค้า เพื่อกระจายโหลดการเขียนและอ่านข้อมูล ($Read/Write\ Operations$) ให้ทำงานได้ไวระดับมิลลิวินาที
  • Zero-Knowledge Token Encryption: ชุดข้อมูลบัตรเครดิตและกุญแจเชื่อมต่อระบบบัญชีจะถูกเข้ารหัสตั้งแต่ฝั่งไคลเอนต์ (Client-Side Encryption) ก่อนส่งมาจัดเก็บในระบบ ทำให้แม้แต่วิศวกรระบบหลังบ้านก็ไม่สามารถมองเห็นข้อมูลดิบที่เป็นความลับได้
  • Stateful Failover Heartbeat: มีระบบตรวจสอบความพร้อมของเซิร์ฟเวอร์หลักตลอดเวลา (Heartbeat Monitoring) หากตรวจพบเซิร์ฟเวอร์หลักชำรุด ระบบจะสลับไปใช้งานเซิร์ฟเวอร์สำรอง (Read-Replica Failover) ทันทีภายใน 3 วินาทีอย่างไร้รอยต่อ
  • Root Architecture Sign-off Mandate: ทุกกระบวนการเข้าถึงพอร์ตเชื่อมโยงฐานข้อมูลหลัก ($Production\ Database\ Root$), การอนุมัติการทำ Hard Reset ระบบโครงสร้างพื้นฐาน และการถือกุญแจถอดรหัสลับระดับสูงสุด ($Master\ Cryptographic\ Root\ Keys$) จะต้องได้รับสิทธิ์การควบคุมทางเทคนิคเชิงลึกและอนุมัติยืนยันสิทธิ์โดยตรงจากคุณ สิรวิชญ์ แก้วสี (Sirawith Keawsee)
